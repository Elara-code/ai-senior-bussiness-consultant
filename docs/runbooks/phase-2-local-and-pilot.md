# 阶段二本地运行与试点手册

## 1. 启动与验证

1. 复制 `.env.example` 为 `.env`，只填写本地或试点专用配置。
2. 执行 `docker compose up -d --build`。
3. 打开 `http://localhost:3000`，并访问 `http://localhost:8000/health` 检查 API。
4. 执行 `./scripts/demo-phase2.sh`，确认项目、材料、需求、方案、审批、提案、交付、知识候选和导出闭环。

默认是 Fake 模式：输出稳定、无外部调用，适合开发、培训和验收。真实项目必须先脱敏并使用独立项目空间。

## 2. 真实模型与 Dify

- 将模型地址、模型名和密钥放在部署平台的 Secret 中，不提交 `.env` 或密钥。
- 使用 OpenAI-compatible 适配器时，先在非生产环境做超时、限流、结构化输出和敏感信息测试。
- Dify 只编排可替换的生成工作流；项目权限、审批状态、版本和审计仍由本系统保存。
- 配置 Dify 地址、应用密钥和回调签名后，先用脱敏材料执行一次影子运行，再切换流量。
- 切回 Fake 模式时移除外部提供商配置并重启 API；已有业务快照不受影响。

## 3. 备份与恢复

- 每日备份 PostgreSQL；对象存储启用版本管理并定期核对清单。
- 恢复演练必须同时验证数据库、对象内容、成果版本和引用目标。
- Redis 事件用于运行进度，不作为业务事实的唯一存储。
- 重试失败 Agent 前先查看最后事件；重试沿用原输入但产生可审计的新运行状态。导出失败可安全重试。

## 4. 常见故障

- `401`：令牌缺失、签名密钥不一致或过期。开发环境重新生成令牌；试点环境检查 SSO 配置，禁止绕过认证。
- `409`：通常是幂等键被其他请求使用或版本已变化。每个项目使用唯一幂等键；编辑冲突时刷新并基于最新 `ETag` 重新提交。
- 超时：确认模型/Dify 可用、输入大小和步骤上限；不要盲目连续重试。保留运行 ID 后从 SSE 最后事件恢复。
- 页面无项目：检查 Web 到 API 的内部地址和双方开发签名密钥是否一致。
- Docker 健康检查失败：先查看 API、PostgreSQL、Redis、MinIO 的健康状态，再重建单个失败服务。

## 5. 安全边界

- 不上传未授权个人信息、合同原文、生产凭据或未脱敏客户数据。
- 每次导出前核对引用、承诺、金额、交期与审批快照。
- 不在日志、截图、工单和演示脚本中粘贴令牌或密钥。
- 试点账号遵循最小权限；跨组织访问应返回 `404`，避免泄露资源存在性。
- 发现敏感泄露时立即停止外部模型调用、撤销相关密钥、隔离导出并记录审计事件。
